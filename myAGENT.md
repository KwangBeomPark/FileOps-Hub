# myAGENT.md - 공통 개발 원칙 문서

본 문서는 프로젝트에서 코드를 작성하고 검증할 때 지켜야 하는 범용적인 개발 원칙과 행위 지침을 정의합니다.

## 1. 소통 원칙
- 사용자와의 모든 소통은 기본적으로 **한국어**로 명확하고 정중하게 진행합니다.
- 기술적인 세부 사항은 명확하고 이해하기 쉬운 문장으로 설명하며, 과장이나 자랑 섞인 표현(예: "완벽하게", "결함 없이" 등)을 피하고 겸손한 톤을 유지합니다.

## 2. 코드 작성 전 분석 절차
- 소스 코드를 수정하거나 새로운 기능을 개발하기 전, 현재 제공된 파일 구조와 기존 코드의 작동 방식을 철저히 분석합니다.
- 프로젝트 루트의 `myAGENT.md`와 `AI_CODE_MAP.md`를 반드시 최우선으로 정독하여 프로젝트 맥락을 파악합니다.

## 3. 코딩 표준 및 네이밍 가이드
- **언어 및 스타일**: Python의 PEP 8 스타일 가이드를 준수합니다.
- **클래스/함수 네이밍**: 
  - 클래스명은 `PascalCase`를 사용합니다.
  - 함수명 및 변수명은 `snake_case`를 사용합니다.
  - 의미가 명확하고 직관적인 이름을 부여하며, 약어 사용은 최소화합니다.
- **주석 및 문서화**: 
  - 핵심 함수와 클래스에는 Docstring을 명확히 작성합니다.
  - 복잡한 알고리즘이나 비즈니스 로직에는 코드 이해를 돕기 위한 인라인 주석을 작성합니다.
  - 기존의 주석과 Docstring 중 수정 대상과 무관한 부분은 무조건 유지(보존)합니다.

## 4. 금지 패턴 및 제한 사항
- **대규모 리팩토링 금지**: 사용자가 명시적으로 요청하지 않은 전면적인 코드 구조 변경은 절대 금지합니다. 기존 기능을 깨뜨리지 않는 '최소 변경 원칙(Principle of Least Change)'을 최우선으로 삼습니다.
- **하드코딩 지양**: 파일 경로, API 키, 설정값 등은 가능한 설정 파일(JSON 등)이나 환경 변수로부터 로드되도록 설계합니다.

## 5. 파일 접근 및 안전 규칙
- 파일 수정 시에는 전체 파일을 덮어쓰는 것보다 필요한 부분만 정밀하게 수정하는 도구(예: `replace_file_content` 또는 `multi_replace_file_content`)를 우선 사용합니다.
- 중요 데이터를 삭제하거나 시스템 설정을 영구적으로 변경할 수 있는 안전하지 않은 명령은 절대 사용자 승인 없이 실행하지 않습니다.
- **원자적 파일 기록 (Atomic Write)**: 앱의 핵심 설정 JSON 등을 물리 디스크에 쓸 때는 먼저 임시 파일(`.tmp`)에 전체를 쓴 뒤 `os.replace`로 원자적으로 교체하여 정전이나 비정상 종료 시 설정 데이터가 깨지는 것을 원천 방어합니다.
- **네이티브 자원 및 좀비 방어**: ctypes 외부 DLL 메모리(`LocalFree`) 또는 외부 브라우저 프로세스(`browser.close()`)를 다룰 때는 예외가 발생해도 반드시 반환되도록 `try...finally` 블록을 통하여 안전 릴리즈 처리를 이행합니다.
- **플랫폼 예외 및 읽기 전용 대응**: Windows DPAPI 등 OS 종속 모듈은 타 환경을 고려하여 Fallback 로직을 탑재하고, 파일 복사 시 대상 파일의 '읽기 전용' 속성에 따른 `PermissionError`를 대비해 사전에 쓰기 속성을 복구(`os.chmod`)하고 수행합니다.

## 6. 단계별 자가 검증 및 이행 루프 (가장 중요)
- 개발은 `task.md`에 명시된 순서(Phase)에 따라 엄격히 한 단계씩 수행합니다.
- **도구 폴더(tools/) 기반 누적 검증**: 
  - 모든 검증은 루트의 `tools/` 폴더 내에 테스트 스크립트를 작성하여 수행합니다.
  - 작성된 테스트 스크립트는 일회성으로 방치하지 않고, 새로운 Phase가 진행될 때마다 지속적으로 보완 및 통합하여 **이전 기능의 정상 여부까지 누적 검증(Regression Testing)할 수 있도록 재사용 가능한 구조로 설계**합니다.
- 각 단계의 코드를 작성한 직후, 다음 단계를 진행하기 전에 반드시 해당 단계의 자가 검증 스크립트(예: `tools/test_*.py`)를 터미널에서 실행하여 테스트 결과가 **성공(PASS)**하는지 검증합니다.
- 검증 스크립트 실행 결과를 실제 출력 로그와 함께 사용자에게 공유하고, `task.md`를 업데이트합니다.
- 테스트가 실패할 경우 다음 단계로 진행하는 것을 즉시 중단하고 오류를 디버깅합니다. 모든 테스트가 완전히 통과(PASS)된 것이 입증된 후에만 다음 단계의 개발로 이행합니다.


## 7. 문서화 및 테스트 원칙
- 작업 완료 후, 변경 사항 및 확인 사항을 `walkthrough.md`에 기록하고 필요한 경우 사용자에게 실행 결과를 설명합니다.
- 새로 추가되는 복잡한 유틸리티나 알고리즘에는 필요에 따라 단위 테스트를 작성하여 검증 가능하도록 합니다.

## 8. 회사 Windows 환경 기준
- 관리자 권한을 전제로 구현하지 않습니다.
- 네트워크 드라이브, OneDrive, SharePoint는 지연·일시적 연결 해제·파일 잠금이 발생할 수 있으므로 원본 보존과 재시도 가능한 실패 보고를 우선합니다.
- Office COM, Windows DPAPI, 파일 생성 시각처럼 Windows 전용 기능은 명시적으로 격리하고 자원을 `finally`에서 해제합니다.
- 경로는 `os.path`/`pathlib`로 정규화하고 공백, 한글, 긴 경로, 읽기 전용 파일을 고려합니다.

## 9. 언어별 기본 규칙
- Python 3.14와 PEP 8을 기준으로 하며 파일 입출력에는 인코딩을 명시합니다.
- 예외를 무조건 삼키지 않고 사용자 메시지와 진단 로그를 구분합니다.
- GUI 스레드에서 긴 파일·네트워크 작업을 수행하지 않으며 `QThread`의 종료와 취소 상태를 검증합니다.
- 새 의존성은 `requirements.txt`와 빌드 스펙, `AI_CODE_MAP.md`에 함께 반영합니다.
